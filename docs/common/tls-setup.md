# PrimeCart Local TLS Setup

This document describes the local TLS setup used by PrimeCart. It is intended
for learning and local development. The certificates are generated by `mkcert`;
no certificate purchase is required.

## Final architecture

TLS is enabled on the browser-facing applications:

```text
Browser
  |
  +-- HTTPS --> React/Vite UI       https://localhost:5173
  +-- HTTPS --> API Gateway         https://localhost:8181
  +-- HTTPS --> Keycloak            https://localhost:8443

API Gateway
  +-- HTTP --> Product Service      http://localhost:8081
  +-- HTTP --> Order Service        http://localhost:8082
  +-- HTTP --> Cart Service         http://localhost:8083
  +-- HTTP --> Inventory Service    http://localhost:8084
  +-- HTTP --> Payment Service      http://localhost:8085
```

Internal service-to-service API calls remain HTTP for this learning setup.
Services still connect to Keycloak over HTTPS when they validate JWTs or obtain
client-credentials tokens.

## Local paths used in this repository

Absolute paths are convenient on this development machine but are not portable.
Other developers must replace them with paths appropriate to their machines.

## 1. Install and verify mkcert

Install `mkcert` on macOS:

```bash
brew install mkcert
mkcert -install
```

Show the local CA directory:

```bash
mkcert -CAROOT
```

Verify that macOS trusts the local CA:

```bash
security verify-cert \
  -c "/Users/dadaramjadhav/Library/Application Support/mkcert/rootCA.pem" \
  -p ssl
```

Expected output includes:

```text
certificate verification successful
```

Certificate Transparency warnings are expected for a private development CA.
Public CT logs do not contain locally generated mkcert certificates.

## 2. Generate the localhost certificate

From the repository root:

```bash
mkdir -p /Users/dadaramjadhav/primecart/infra/tls

mkcert \
  -cert-file /Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem \
  -key-file /Users/dadaramjadhav/primecart/infra/tls/localhost-key.pem \
  localhost 127.0.0.1 ::1
```

The certificate is used by the UI, API Gateway, and Keycloak. The private key
must remain local and must never be committed.

Verify the certificate chain:

```bash
openssl verify \
  -CAfile "/Users/dadaramjadhav/Library/Application Support/mkcert/rootCA.pem" \
  /Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem
```

Expected:

```text
/Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem: OK
```

## 4. Enable HTTPS in Keycloak

The Keycloak Compose service publishes both HTTP and HTTPS while the migration
is in progress:

```yaml
services:
  keycloak:
    image: quay.io/keycloak/keycloak:latest
    container_name: keycloak
    command: start-dev

    ports:
      - "8080:8080"
      - "8443:8443"

    volumes:
      - /Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem:/opt/keycloak/conf/tls/localhost-cert.pem:ro
      - /Users/dadaramjadhav/primecart/infra/tls/localhost-key.pem:/opt/keycloak/conf/tls/localhost-key.pem:ro

    environment:
      KC_HTTPS_CERTIFICATE_FILE: /opt/keycloak/conf/tls/localhost-cert.pem
      KC_HTTPS_CERTIFICATE_KEY_FILE: /opt/keycloak/conf/tls/localhost-key.pem
```

The database and bootstrap administrator settings are omitted here. Store real
passwords outside the Compose file. The bootstrap administrator variables only
create an administrator when the Keycloak database is initially created; they
do not reset an existing administrator password.

Start or recreate Keycloak:

```bash
docker compose \
  -f /Users/dadaramjadhav/primecart/infra/keycloack/docker-compose-keycloak.yml \
  up -d
```

Inspect startup logs:

```bash
docker compose \
  -f /Users/dadaramjadhav/primecart/infra/keycloack/docker-compose-keycloak.yml \
  logs -f keycloak
```

The log should show both listeners:

```text
http://0.0.0.0:8080
https://0.0.0.0:8443
```

Open the Administration Console at:

```text
https://localhost:8443/admin/master/console/
```

For a stable project setup, pin the Keycloak image to a tested version instead
of using `latest`.

## 5. Update the Keycloak React client

In the `primecart` realm, open the `primecart-react` client and configure:

```text
Valid redirect URIs: https://localhost:5173/*
Web origins:         https://localhost:5173
```

Keep the client public because a browser application cannot safely hold a
client secret.

The access token used by the Gateway must include the expected audience:

```text
primecart-api
```

If necessary, add an Audience mapper to a client scope assigned to
`primecart-react`:

```text
Name:                     primecart-api-audience
Included Client Audience: primecart-api
Add to access token:      On
Add to ID token:          Off
```

## 6. Enable HTTPS in the React/Vite UI

The UI `.env` uses HTTPS endpoints:

```dotenv
VITE_API_BASE_URL=https://localhost:8181
VITE_KEYCLOAK_URL=https://localhost:8443
VITE_KEYCLOAK_REALM=primecart
VITE_KEYCLOAK_CLIENT_ID=primecart-react
```

Vite reads its environment file only at startup, so restart Vite after changing
these values.

The relevant `vite.config.js` configuration is:

```javascript
import fs from "node:fs"
import { fileURLToPath, URL } from "node:url"

const certificatePath = fileURLToPath(
  new URL("../infra/tls/localhost-cert.pem", import.meta.url),
)

const privateKeyPath = fileURLToPath(
  new URL("../infra/tls/localhost-key.pem", import.meta.url),
)

export default defineConfig({
  server: {
    https: {
      cert: fs.readFileSync(certificatePath),
      key: fs.readFileSync(privateKeyPath),
    },
  },
})
```

The development Content Security Policy permits the required HTTPS and WSS
connections:

```text
connect-src 'self' https://localhost:8181 https://localhost:8443 wss://localhost:5173
frame-src https://localhost:8443
form-action 'self' https://localhost:8443
```

Start the UI:

```bash
cd /Users/dadaramjadhav/primecart/primecart-ui
npm run dev
```

Open:

```text
https://localhost:5173
```

## 7. Enable HTTPS in API Gateway

The Gateway uses Spring Boot's PEM certificate properties:

```yaml
server:
  port: 8181
  ssl:
    enabled: true
    certificate: file:/Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem
    certificate-private-key: file:/Users/dadaramjadhav/primecart/infra/tls/localhost-key.pem
```

An environment-based version is preferable when sharing the project:

```yaml
server:
  port: 8181
  ssl:
    enabled: true
    certificate: ${TLS_CERT_PATH}
    certificate-private-key: ${TLS_KEY_PATH}
```

Corresponding environment variables:

```bash
export TLS_CERT_PATH="file:/Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem"
export TLS_KEY_PATH="file:/Users/dadaramjadhav/primecart/infra/tls/localhost-key.pem"
```

Gateway CORS permits the HTTPS UI origin:

```yaml
spring:
  cloud:
    gateway:
      server:
        webflux:
          globalcors:
            cors-configurations:
              '[/**]':
                allowedOrigins:
                  - "https://localhost:5173"
```

The downstream route URLs remain HTTP in this setup.

## 8. Use the HTTPS Keycloak issuer everywhere

Every JWT resource server must use the issuer contained in newly issued tokens:

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://localhost:8443/realms/primecart
```

This applies to the API Gateway and every microservice that validates a bearer
token directly.

Issuer matching is exact. These are different issuers:

```text
http://localhost:8080/realms/primecart
https://localhost:8443/realms/primecart
```

A token issued with the first value is rejected by a resource server configured
with the second value.

## 9. Update OAuth client-credentials token endpoints

Services that obtain tokens for service-to-service calls must request those
tokens through HTTPS:

```yaml
spring:
  security:
    oauth2:
      client:
        provider:
          keycloak:
            token-uri: https://localhost:8443/realms/primecart/protocol/openid-connect/token
```

This change is currently required by:

- Payment Service, which calls Order Service.
- Order Service, which calls Inventory Service.

If a service requests its token through `http://localhost:8080`, Keycloak can
issue a token whose `iss` claim uses HTTP. A target service expecting the HTTPS
issuer then returns `401 Unauthorized`.

Restart a service after changing `token-uri` so its OAuth client manager cannot
reuse a cached HTTP-issued access token.

## 10. Create a Java truststore

macOS and browsers trust the mkcert CA after `mkcert -install`, but Java processes
may still require an explicit truststore.

Create a PKCS12 truststore:

```bash
keytool -importcert \
  -noprompt \
  -alias primecart-local-ca \
  -file "/Users/dadaramjadhav/Library/Application Support/mkcert/rootCA.pem" \
  -keystore "/Users/dadaramjadhav/primecart/infra/tls/primecart-truststore-v2.p12" \
  -storetype PKCS12 \
  -storepass changeit
```

Verify the entry and password:

```bash
keytool -list \
  -alias primecart-local-ca \
  -keystore "/Users/dadaramjadhav/primecart/infra/tls/primecart-truststore-v2.p12" \
  -storetype PKCS12 \
  -storepass changeit
```

Expected entry type:

```text
trustedCertEntry
```

If a truststore already exists with an unknown password, create a new file
rather than overwriting it. The password `changeit` is acceptable only for this
local learning setup.

## 11. Start Java services with the truststore

Set `JAVA_TOOL_OPTIONS` in the same terminal used to launch a service:

```bash
export JAVA_TOOL_OPTIONS="\
-Djavax.net.ssl.trustStore=/Users/dadaramjadhav/primecart/infra/tls/primecart-truststore-v2.p12 \
-Djavax.net.ssl.trustStorePassword=changeit \
-Djavax.net.ssl.trustStoreType=PKCS12"
```

For example, start the Gateway:

```bash
cd /Users/dadaramjadhav/primecart/api-gateway
mvn spring-boot:run
```

Java should report that it picked up `JAVA_TOOL_OPTIONS`.

Use the same truststore options for every service that communicates directly
with HTTPS Keycloak. In the current design this includes all JWT-validating
resource servers and the services using OAuth client credentials.

### IntelliJ run configurations

Shell environment exports may not be visible to a service launched from
IntelliJ. Add the following to each applicable run configuration under
**VM options**:

```text
-Djavax.net.ssl.trustStore=/Users/dadaramjadhav/primecart/infra/tls/primecart-truststore-v2.p12 -Djavax.net.ssl.trustStorePassword=changeit -Djavax.net.ssl.trustStoreType=PKCS12
```

Completely stop and restart the application after adding the options.

## 13. JWT checks

For this project, a valid UI access token should contain values similar to:

```json
{
  "iss": "https://localhost:8443/realms/primecart",
  "aud": ["primecart-api"],
  "realm_access": {
    "roles": ["PRODUCT_READ"]
  }
}
```

Interpret common responses as follows:

- `401 Unauthorized`: token signature, issuer, audience, or expiry validation
  failed.
- `403 Forbidden`: the token was authenticated, but the required role or
  authority is missing.

Never paste a complete access token into logs, chat, tickets, or public JWT
decoding websites. Decode only the payload locally when debugging.

## 14. Troubleshooting

### PKIX path building failed

Error:

```text
PKIX path building failed
unable to find valid certification path to requested target
```

Check:

1. The truststore contains the current mkcert root CA.
2. The truststore password is correct.
3. `javax.net.ssl.trustStoreType` is `PKCS12`.
4. The service process received the JVM truststore options.
5. The service was fully restarted after adding the options.

Verify the certificate and CA belong to the same chain:

```bash
openssl verify \
  -CAfile "/Users/dadaramjadhav/Library/Application Support/mkcert/rootCA.pem" \
  /Users/dadaramjadhav/primecart/infra/tls/localhost-cert.pem
```

### Login succeeds but the UI receives 401

After changing from HTTP to HTTPS, the browser may retain a token with the old
HTTP issuer. Close all application and Keycloak tabs, clear site data for these
origins, and log in again:

```text
http://localhost:8080
https://localhost:8443
http://localhost:5173
https://localhost:5173
```

The new token's `iss` must be:

```text
https://localhost:8443/realms/primecart
```

Its `aud` must include:

```text
primecart-api
```

### Payment succeeds until its call to Order Service

If Payment Service receives the UI request but Order Service returns `401`,
inspect the outgoing service token's issuer without logging the token itself.
The Payment Service `token-uri` must use HTTPS, and Payment Service must be
restarted to discard any cached HTTP-issued token.

### Keycloak reports invalid redirect URI

Ensure the `primecart-react` client contains:

```text
Valid redirect URIs: https://localhost:5173/*
Web origins:         https://localhost:5173
```

### Keycloak administrator is logged out after a restart

Close old Keycloak tabs and clear site data for ports `8080` and `8443` before
trying again. Old refresh tokens may no longer be valid after recovery or
session changes.

### Netty macOS DNS warning

The following warning is not the cause of JWT `401` responses:

```text
Unable to load MacOSDnsServerAddressStreamProvider
```

It indicates that Netty is using the system DNS fallback.

## 15. Security cleanup

- Do not log complete UI or service access tokens.
- Revoke sessions if a live token is accidentally exposed.
- Keep private keys and truststores out of Git.
- Move client secrets and passwords to environment variables or a secret store.
- Do not use `changeit` as a production truststore password.
- Do not use `start-dev` for production Keycloak.
- Pin container image versions.
- Use certificates from a trusted organizational or public CA in production.
- When ready to enforce HTTPS-only Keycloak access, remove the published `8080`
  port and confirm that no configuration still references it.
